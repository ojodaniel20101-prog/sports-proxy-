# Railway HLS Proxy - Deployment Guide

## Quick Start (3 steps)

### Step 1: Create Railway Project
```bash
# Install Railway CLI
npm install -g @railway/cli

# Login
railway login

# Create project
railway init --name my-hls-proxy
```

### Step 2: Deploy
```bash
# Add these files to your repo:
# - railway_hls_proxy.js
# - package.json
# - .gitignore

git add .
git commit -m "Initial HLS proxy"
git push railway main
```

### Step 3: Set Environment Variables
In Railway dashboard, set:
- `SPORTS_API_URL` = `https://h5-api.aoneroom.com`
- `API_AUTH_TOKEN` = (your API auth token if required)

## How It Works

```
User Browser          Railway Proxy          Aoneroom API/CDN
     |                      |                        |
     |--- GET /api/matches ->|                        |
     |                      |--- GET /matches ------->|
     |                      |<-- Stream URLs          |
     |                      |   (signed for Railway IP)|
     |<-- Match data -------|                        |
     |                      |                        |
     |--- GET /proxy/stream?url=...                  |
     |                      |                        |
     |                      |--- GET playlist.m3u8 ->|
     |                      |<-- .m3u8 (403 avoided!)|
     |                      |                        |
     |<-- Rewritten playlist|                        |
     |   (segments point    |                        |
     |    to /proxy)        |                        |
     |                      |                        |
     |--- GET /proxy?url=segment.ts                 |
     |                      |                        |
     |                      |--- GET segment.ts ---->|
     |                      |<-- Video data          |
     |<-- Video segment ----|                        |
```

## Why This Works

The `sign=` parameter is an **IP-locked MD5 hash** generated by Shalltry CDN.
When you call the API from Server A but play from Server B:
- Token is signed for Server A's IP
- Server B has different IP → 403 Forbidden

**Solution**: Both API call AND stream fetch must come from the **same IP**.
The Railway proxy ensures this by:
1. Calling the API from Railway's IP
2. Getting tokens signed for Railway's IP
3. Fetching streams from Railway's IP (same IP = token valid)
4. Relaying to clients

## Architecture Notes

- **Playlists are cached** for 10 seconds (they update frequently in live streams)
- **Segments are NOT cached** by default (they're large video files)
- The proxy rewrites `.m3u8` playlists to route `.ts` segments through itself
- All query parameters (including `sign=` and `t=`) are preserved

## Limitations

1. **Bandwidth**: Your Railway server becomes a bandwidth bottleneck
2. **Latency**: Added hop between client and stream
3. **Cost**: Video streaming uses significant bandwidth
4. **Scale**: Single Railway instance may struggle with many concurrent users

## Alternative: Client-Side Approach

If you want to avoid proxy bandwidth, have the client call the API directly:

```javascript
// In browser/app
const response = await fetch('https://h5-api.aoneroom.com/your-endpoint');
const matches = await response.json();
// Stream URLs are signed for the USER'S IP - works directly!
player.src = matches[0].playPath; // Plays directly, no proxy needed
```

This only works if:
- The API supports CORS (for browser requests)
- You're okay exposing the API endpoint to clients
